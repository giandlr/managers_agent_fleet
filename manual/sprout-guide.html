What Is Managers' Agent Fleet?

This is a system that lets you build production-quality apps by describing what you want in plain English. You never touch code. Quality, security, and testing happen automatically behind the scenes.

💬

You Describe

Tell Claude Code what you want in plain language. "I need a dashboard that shows team performance metrics."

🤖

AI Builds

Claude Sonnet writes the code, writes the tests, and follows every engineering best practice automatically.

✅

Quality Is Automatic

Security checks, tests, and an independent AI code reviewer ensure everything works correctly — when you're ready to share.

Think of it as your invisible engineering team. You build freely during development, and when you're ready to share or ship, the full quality system kicks in automatically. You focus on what to build, not how to build it.

Build Freely, Deploy Safely

The system has two modes. You're always in Build mode by default. Deploy mode activates when you're ready to share.

Build Mode (default)

You build freely

This is your creative workspace. Only security issues are blocked — everything else, Claude handles quietly in the background.

  • Claude writes code and auto-fixes quality issues as it goes
  • Claude explains what it did in plain English after each step
  • No pipeline runs, no formal reviews — just fast iteration
  • Security problems are still caught immediately (passwords in code, dangerous commands)
Claude says "I built the team dashboard with a bar chart showing monthly performance. I added a loading indicator so users see something while the data loads, and I made sure each manager can only see their own team's data."
Deploy Mode (when you're ready)

Quality checks activate

When you say "ship it" or "share with my team," Claude switches to deploy mode and runs the appropriate level of checks.

  • Claude asks: "Is this just for you, your team, or external users?"
  • Runs the right level of validation for your answer
  • If everything passes, a changelog entry is created automatically
  • After the pipeline finishes, you're back in build mode
Claude says "I'm running checks to make sure this is ready for your team. Verifying security... checking that all features work... making sure errors are handled properly. Everything passed — it's ready to share."

Three Levels of Validation

When you deploy, Claude asks who it's for and runs the right level of checks. Start simple and level up as your project grows.

MVP

"Just me testing"

A quick check to make sure nothing is broken or insecure. Takes about a minute.

  • Security scan — no passwords or secrets in code
  • App starts without errors
  • The main feature works end-to-end
Say: "Let me test this" or "Quick check"
Team

"Sharing with colleagues"

Validates that the app is reliable enough for other people to use. Takes a few minutes.

  • Everything in MVP, plus:
  • All tests pass with reasonable coverage
  • Errors are handled gracefully (no blank screens)
  • Login and permissions work correctly
Say: "Share with my team" or "Ship it to the team"
Production

"External users"

Full enterprise validation. Performance testing, independent code review, everything. Takes 5–10 minutes.

  • Everything in Team, plus:
  • Full test suite with high coverage
  • Performance validated — fast response times
  • Independent AI code review (Opus)
  • All enterprise features present
Say: "Ship to production" or "Ready for external users"
You don't need to start at production level. Build your MVP, test it yourself, share it with your team, and graduate to production when you're ready. The system grows with you.

How It Works

Layers of protection work silently. In build mode, only security guards are active. In deploy mode, everything kicks in.

Layer 1 — Rules

Memory & Standards

Claude remembers the project's rules: which tech to use, how to structure code, what security patterns to follow. These rules are always loaded — you never need to repeat them.

Layer 2 — Guards

Real-Time Protection

Every time Claude writes code, security checks run instantly. In build mode, only dangerous patterns are blocked. In deploy mode, full quality checks activate. All messages are in plain English.

Layer 3 — Agents

Specialist Workers

Dedicated AI agents handle specific jobs in parallel: one builds the backend, another the frontend, others run tests and security scans — all at the same time.

Layer 4 — Pipeline

Deploy Validation

When you deploy, the pipeline runs at your chosen gate level (MVP, Team, or Production). Only the checks appropriate for that level are executed.

Layer 5 — Independent Review (Production only)

Opus Code Review

At the production gate level, a separate senior AI (Claude Opus) reviews the code with zero knowledge of the development session. Like an independent auditor checking every change before it goes live.

Start With Discovery

When you start a brand-new project, Claude asks a few quick questions before building. This takes about 2 minutes and saves hours of rework.

💬

Round 1: Big Picture

"What kind of app is this?" and "Who will use it?" — Claude gives you clickable options so you don't have to type.

✅

Round 2: Features

Based on your answer, Claude suggests features as a checklist. Pick what you want, skip what you don't. Claude also challenges: "Is there anything I'm missing?"

🔒

Round 3: Access

"Should everyone see everything, or only their own stuff?" — This determines the security rules Claude puts in place.

After 2–3 quick rounds, Claude writes a project brief and asks "Does this look right?" Once you confirm, building starts immediately. If your initial request is already detailed, Claude skips straight to the brief.

Installation — One-Time Setup

Before your first project, run a single setup command. After that, you never touch it again.

1

Create a new project folder and initialize git

mkdir my-app && cd my-app && git init
2

Copy the toolkit zip into your project and unzip it

cp /path/to/managers-agent-fleet.zip .
unzip managers-agent-fleet.zip

This creates a managers-agent-fleet/ folder inside your project with all the toolkit files.

3

Install the toolkit into your project

bash managers-agent-fleet/install.sh

This copies the toolkit files into the right locations, sets up your .gitignore so toolkit files are never committed, and cleans up the zip and installer folder automatically.

4

Bootstrap your app

bash scripts/sprout-bootstrap.sh

This creates your entire app structure, installs everything, starts the database, backend, and frontend. When it finishes, your app is already running at localhost:3000.

5

Fill in your app name

Open CLAUDE.md in the project root and replace [APP_NAME] with your application's name and a one-sentence description.

6

Open Claude Code and start building

claude
That's it. You're in build mode by default — build freely, and say "ship it" when you're ready to share.

The Opus Review — Your Independent Auditor

At the production level, after all tests pass, a separate senior AI reviews the code from scratch with zero bias.

✅

APPROVED

Everything looks great. No issues found. All enterprise features are present. Ready for production.

🟡

APPROVED WITH CONDITIONS

Code is good and functional. Some features needed for production haven't been built yet. The number will decrease as you keep building.

🛑

CHANGES REQUIRED

Problems were found that must be fixed: security issues, broken logic, or missing critical features. Claude will fix these automatically.

Opus has zero knowledge of the development process. It only sees the code changes and test results. This is intentional — it eliminates bias and catches issues the builder might miss. Like an external auditor reviewing financial statements. This review only runs at the production gate level.

Enterprise Features — Built In, Not Bolted On

Every app you build automatically includes enterprise-grade features. They're split into two tiers.

Required Tier 1 — Must Have

The production pipeline will not pass until all of these exist.

Login & Security: Email/password, Google SSO, multi-factor auth, role-based access (Admin, Manager, Member, Viewer), invite-only sign-up

Reliability: Health check endpoint, error pages (never shows raw errors), rate limiting, idempotent operations

User Experience: Pagination everywhere, loading spinners, error messages, empty states, confirmation dialogs, form validation

Security: HTTPS, security headers, restricted access origins, no secrets in frontend

Accessibility: Mobile responsive, keyboard navigation, color contrast compliance

Pre-Production Tier 2 — Before Go-Live

Opus flags these as advisories. Must be complete before real users access the app.

Notifications: In-app, email, push, user preferences, digest mode

Compliance: Audit trail, data export, retention policies, consent tracking, right to erasure

Performance: Caching, lazy loading, optimistic updates, search with debounce

Resilience: Maintenance mode, feature flags

Extras: PWA support, error tracking, admin dashboard, help tooltips, feedback widget

When Something Is Blocked

In build mode, only security issues cause blocks. In deploy mode, quality checks can also block. Either way, messages are always in plain English.

Always blocked (build + deploy)

Passwords or secrets in code
Passwords in code can get leaked. They must be stored securely.

Dangerous commands
Commands that could destroy data or force-push over other people's work.

Internal keys in the wrong place
Admin-level keys must never appear in user-facing code.

Blocked only in deploy mode

Missing loading or error states
Users shouldn't stare at blank screens. Every action must show its status.

Code quality issues
Type errors, formatting problems, and other quality issues that Claude auto-fixes in build mode.

What you need to do

Absolutely nothing.

When something is blocked, Claude sees the message, understands the problem, and fixes it automatically. All block messages are in plain English — no technical jargon.

What you'll see

Claude says "A password was found written directly in the code. I'll move this to an environment variable."
Claude says "This includes a master database key visible to app users. I'll switch to the safe public key for frontend code."

Claude Speaks Plain English

Claude always explains what it did and why — in language anyone can understand. No technical jargon, ever.

What Claude tells you

After building a feature "I added a loading spinner so users see something while data loads. I also added an error message in case the server is unavailable, and an empty state for when there's no data yet."
After adding security "I added security rules so each user only sees their own data. Admin users can see everything, but regular members can only access records they created."
After fixing a problem "I found and fixed the issue causing the blank screen on mobile. The page was trying to load before it had all the information it needed. Now it waits for the data first."

What Claude does NOT say

"Ran ESLint with --max-warnings=0, fixed 3 violations. Ran mypy, resolved 2 type errors in auth_service.py."

Tool names and technical flags are never shown to you.

"Added RLS policy with auth.uid() = user_id predicate on the invoices table."

Database syntax is translated to plain English.

Instead, Claude says:

"I verified the code has no errors and fixed a few small issues." / "I added security rules so each user only sees their own invoices."

Releases & Deployment

Two optional actions you can take when you're ready to ship.

When ready to version

Cutting a Release

When a set of features is complete and you want to call it a version, tell Claude:

Run bash .claude/scripts/tag-release.sh v1.0.0

This freezes the current changelog into a named version and creates a tag. Version numbers work like this:

  • v1.0.0 → v1.0.1 — Small fix (bug, typo)
  • v1.0.0 → v1.1.0 — New feature added
  • v1.0.0 → v2.0.0 — Major change
After deployment

Recording a Deployment

After the app goes live (your tech team handles the actual deployment), record it:

Run bash .claude/scripts/mark-deployed.sh production

This updates the dev-log and deployment state so everyone knows what's live and when it went out.

  • Use staging for test environments
  • Use production for the real, live app
  • The changelog records who deployed and when

Both commands are optional. The system works fine without them — they're for when you want formal version tracking.

Tips for Working With Claude

A few things that make the experience smoother.

Do

  • Be specific about what you want — "Add a dashboard showing monthly sales by region" works better than "Add analytics"
  • Say when something looks wrong — "That chart should be a bar chart, not a pie chart" helps Claude fix it immediately
  • Start small and iterate — Build one feature at a time. Each one gets tested and secured as you go.
  • Close and reopen Claude if things feel stuck — Fresh sessions pick up where you left off thanks to crash recovery

Don't

  • Don't ask Claude to skip safety checks — They exist to protect your data and users
  • Don't worry about the terminal — Claude handles commands, servers, and databases for you
  • Don't try to read the code — Unless you want to! Everything is explained in plain English
  • Don't hesitate to change your mind — "Actually, make it a table instead" is perfectly fine at any stage
Claude remembers your project context between sessions. If your terminal closes unexpectedly, just reopen Claude Code and it will offer to pick up where you left off.
